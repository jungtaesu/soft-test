# React + TypeScript + Vite

## 과제 개요

본 과제는 Vite + React + TypeScript 환경에서 진행했습니다.
원본 프로젝트 레포지토리가 제공되지 않아, 과제 수행에 필요한 최소 범위 내에서 mock 데이터와 더미 컴포넌트를 구성했습니다.

# MISSION 1 — 레거시 코드 진단 및 리팩토링

## 개요

주어진 `PostListPage` 코드를 기준으로 렌더링 관점의 성능 저하 원인을 분석하고, AI 제안과 직접 판단한 내용을 함께 반영해 리팩토링했습니다.

---

## [원인 1] 서로 다른 도메인의 비동기 로직이 하나의 effect에 묶여 있음

### 문제 설명

기존 코드는 게시글 조회와 사용자 조회를 하나의 `useEffect` 내부에서 함께 처리하고 있었습니다.
이 구조에서는 `categoryId` 변경 시 게시글만 다시 조회하면 되는 상황에서도 사용자 정보 조회가 함께 실행될 수 있어, 불필요한 fetch와 상태 업데이트 가능성이 있었습니다.

또한 실제 JSX에서 사용되지 않는 `loading` state가 남아 있어 상태 흐름만 복잡하게 만들고 있었습니다.

### AI가 제안한 해결책

별도 제안 없음.

### 내가 채택한 방식 및 이유

* 게시글 조회와 사용자 조회를 도메인별로 분리해 각각의 `useEffect`로 관리
* 실제로 사용하지 않는 `loading` state 제거

이렇게 수정한 이유는 각 effect의 책임을 명확히 하고, 불필요한 상태 업데이트 가능성을 줄이기 위해서입니다.
렌더 횟수 자체보다도, 의존성과 책임이 더 명확한 구조를 만드는 것이 더 중요하다고 판단했습니다.

### 채택하지 않은 방식이 있다면, 그 이유

`loading` state를 유지한 채 향후 로딩 UI를 붙이는 방식도 가능했지만, 현재 과제 범위에서는 실제 사용하지 않는 상태를 남겨두는 것보다 제거하는 편이 더 명확하다고 판단했습니다.

## [원인 2] 값이 변하지 않는 자식 컴포넌트까지 부모 리렌더에 따라 함께 렌더됨

### 문제 설명

`UserProfile`은 카테고리 변경과 직접적인 관련이 없고, 전달되는 `user` 값도 변하지 않는데 부모 컴포넌트인 `PostListPage`가 리렌더될 때 함께 재렌더되고 있었습니다.
실측 기준 `UserProfile`은 카테고리 변경 흐름에서 **4회 렌더링**되고 있었습니다.

### AI가 제안한 해결책

`UserProfile` 컴포넌트를 `React.memo`로 감싸기

### 내가 채택한 방식 및 이유

`UserProfile`에 `React.memo`를 적용했습니다.

적용 후 React Profiler 기준으로 카테고리 변경 시 `UserProfile`은 렌더 커밋에서 제외되었고, **4회 → 0회**로 감소했습니다.
부모 리렌더 자체를 없앤 것은 아니지만, 변화 없는 자식 컴포넌트의 불필요한 렌더 비용을 줄였다는 점에서 의미가 있다고 판단했습니다.

### 채택하지 않은 방식이 있다면, 그 이유

별도의 대안 없이 `React.memo`를 채택했습니다.
이 컴포넌트는 props 변화가 거의 없고, 카테고리 변경과도 직접 관련이 없어 memo 적용 효과가 명확하다고 판단했습니다.

## [원인 3] 함수 prop 참조가 매 렌더마다 새로 생성되어 리스트 아이템 재렌더를 유발함

### 문제 설명

`handleClick` 함수가 부모 렌더링마다 새로 생성되면, `PostItem`을 `memo` 처리하더라도 함수 prop 참조가 달라져 재렌더가 발생할 수 있습니다.
실제로 `PostItem`은 동일 게시글임에도 부모 렌더 시 불필요하게 다시 렌더되는 흐름이 관찰되었습니다.

### AI가 제안한 해결책

`handleClick` 함수에 `React.useCallback` 적용

### 내가 채택한 방식 및 이유

`handleClick`을 `useCallback`으로 메모이제이션했습니다.

적용 후 React Profiler 기준으로 `PostItem` 렌더 횟수는 **4회 → 2회**로 감소했습니다.
이 최적화는 컴포넌트 렌더 자체를 직접 막는 것이 아니라, 자식에게 전달되는 함수 prop의 참조를 안정화하여 `React.memo(PostItem)`이 의미 있게 동작하도록 만든다는 점에서 채택했습니다.

### 채택하지 않은 방식이 있다면, 그 이유

인라인 함수 유지 방식은 구현은 단순하지만, memoized child component의 props 안정성을 해치므로 채택하지 않았습니다.

---

## 추가 검토 사항

### 정렬 로직에 `useMemo`를 적용하지 않은 이유

초기 구조에서는 정렬이 `setPosts` 이후의 렌더에서만 의미 있게 실행되었고, 렌더링 저하의 핵심 원인이 재연산보다 자식 컴포넌트의 불필요한 재렌더에 있다고 판단했습니다.
따라서 Mission 1 단계에서는 `useMemo`를 적용하지 않았습니다.

다만 Mission 2에서 필터 상태가 추가되면서 `sortedAndFilteredPosts`는 여러 state 변화에 따라 반복 계산될 수 있으므로, 해당 시점에는 `useMemo` 적용이 더 타당하다고 판단했습니다.

# MISSION 2 — 트레이드오프 분석과 선택

## 개요

다중 필터링 기능을 구현하면서, 필터 상태를 어디에 둘지와 어떤 방식으로 필터링을 구성할지를 비교했습니다.

---

## [채택한 방식]

### 구현 방식 요약

React의 `useState`로 필터 상태를 관리하고, `useMemo` 내부에서 순수 JS 필터 함수(`filterPosts`)를 통해 카테고리, 날짜 범위, 좋아요 최솟값, 검색어 조건을 AND로 결합했습니다.

### 선택 이유

* 추가 의존성이 없어 번들 크기 증가가 없음
* 구현과 설명이 직관적임
* 단일 화면 기준으로 상태 관리가 단순함
* 과제 범위에 맞는 최소 복잡도로 요구사항을 충족할 수 있음

이번 과제는 단일 페이지에서 필터링 로직을 검증하는 것이 핵심이었기 때문에, 로컬 state 기반 방식이 가장 적절하다고 판단했습니다.

---

## [채택하지 않은 방식]

### 구현 방식 요약

URL search params에 필터 상태를 저장하고, URL 값을 파싱해 필터링에 반영하는 방식

### 치명적 단점 또는 이 프로젝트에 맞지 않는 이유

URL 기반 방식은 새로고침 복원성과 공유 가능한 링크 제공 측면에서는 장점이 있습니다.
하지만 이번 과제에서는 다음과 같은 이유로 과한 설계라고 판단했습니다.

* URL 직렬화/파싱 로직이 추가됨
* 필터 상태와 URL 상태를 함께 관리해야 해 흐름이 복잡해짐
* 부모 컴포넌트 또는 상위 레벨에서 URL 동기화 책임이 커짐
* 단일 화면 과제 규모 대비 구현 복잡도가 불필요하게 증가함

즉, 기능상 장점은 있지만 현재 프로젝트 규모와 과제 목적에는 적합하지 않다고 판단했습니다.

---

## [AI 제안에서 내가 수정한 부분]

### 수정 내용

큰 구조 변경은 하지 않았고, 타입 선언 위치를 컴포넌트 외부로 분리했습니다.

### 수정 이유

필터링 방식 자체는 과제 범위에 적절하다고 판단해 유지했고, 재사용 가능한 타입 정의만 외부로 분리해 가독성과 재사용성을 높였습니다.

예시:

```ts
type PostFormValues = {
  title: string;
  content: string;
  tags: string[];
  likesThreshold: string;
};
```

## MISSION 3 구현 요약

### Validation 엔진

검증 규칙은 JSON-serializable한 순수 객체 형태로 유지하고, 실제 검증 로직은 엔진 내부 함수에서만 처리하도록 구성했습니다.

주요 함수:

* `validateField`
* `validateForm`
* `hasErrors`

### useForm 훅

`useForm(rules, initialValues)` 시그니처를 유지하면서 아래 값을 반환하도록 구현했습니다.

* `values`
* `errors`
* `handleChange`
* `handleSubmit`

### 실제 사용 예시

`PostForm.tsx`에서 제목, 내용, 태그, 좋아요 기준값 입력을 받고, 각 필드별 에러 메시지를 한국어로 출력하도록 구성했습니다.

---

## AI 생성 코드 버그 분석

### 버그 1 — validate 함수 내 템플릿 문자열 오류

* **어느 줄에서 발생하는가?**
  `validate` 함수 내부의 최소/최대 길이 에러 메시지 반환 부분

* **어떤 상황에서 문제가 생기는가?**
  길이 검증 실패 시 `${rule.minLength}`, `${rule.maxLength}`가 실제 값으로 치환되지 않고 문자열 그대로 출력됩니다.

* **수정된 코드:**
  작은따옴표 문자열을 백틱 템플릿 문자열로 수정했습니다.

---

### 버그 2 — handleChange 내부 stale state 가능성

* **어느 줄에서 발생하는가?**
  `handleChange` 함수 내부

* **어떤 상황에서 문제가 생기는가?**
  연속 입력이나 빠른 상태 변경 시 이전 state를 기준으로 덮어쓸 가능성이 있습니다.

* **수정된 코드:**
  `setValues`, `setErrors`를 함수형 업데이트 방식으로 변경했습니다.

---

### 버그 3 — handleSubmit의 이벤트 처리 방식 문제

* **어느 줄에서 발생하는가?**
  `handleSubmit` 함수 내부

* **어떤 상황에서 문제가 생기는가?**
  form submit 시 이벤트를 전달받지 않아 `preventDefault()`를 호출할 수 없고, 실제 form에 연결할 경우 기본 제출 동작이 발생할 수 있습니다.

* **수정된 코드:**
  `handleSubmit = (onSubmit) => (event) => { ... }` 형태로 수정하여 이벤트를 처리하도록 변경했습니다.

---

### 버그 4 — 숫자/배열 규칙 미지원

* **어느 줄에서 발생하는가?**
  기존 `validate` 함수 전체 구조

* **어떤 상황에서 문제가 생기는가?**
  과제에서 요구한 `type: 'number'`, `min`, `max`, `maxCount`, `pattern` 규칙을 처리하지 못합니다.

* **수정된 코드:**
  숫자 규칙과 배열 규칙 분기를 추가해 전체 `postRules`를 지원하도록 보완했습니다.

---

### 버그 5 — 빈 값 판별 로직의 부정확성

* **어느 줄에서 발생하는가?**
  `if (rule.required && !value)` 조건문

* **어떤 상황에서 문제가 생기는가?**
  `0`, 공백 문자열, 빈 배열 등을 정확히 구분하지 못해 의도와 다른 검증 결과를 낼 수 있습니다.

* **수정된 코드:**
  `isEmptyValue` 헬퍼 함수를 추가해 필수값 판별 기준을 명확하게 정리했습니다.
